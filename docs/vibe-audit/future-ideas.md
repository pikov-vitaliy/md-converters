# Идеи в сторону развития

Документ с мыслями, которые **не вошли** в v1.2.0, но могут пригодиться
позже. Не план, не roadmap с датами — просто заметки, чтобы не потерять
контекст при следующем пересмотре.

Если идея дозреет до реализации — её стоит вынести из этого файла в
отдельный этап в `CODEX.md` со своими DoD-критериями и тестами.

## Контекст

`md-converters` используется преподавателем для конвертации документов,
присылаемых студентами, в Markdown для последующей работы через LLM.
Основные входные форматы в этом сценарии: `.docx` (рефераты, курсовые),
`.pdf` (методички, сканы), `.html` (страницы LMS/вики). Реже — `.xlsx`,
`.pptx`. Входные документы — недоверенные.

## 1. Пресет «для проверки студенческих работ»

**Суть:** скрипт-обёртка `tomd-student` (или флаг `--preset student`),
который по дефолту выставляет согласованный набор опций под типовой
сценарий преподавателя: sandbox включён, жёсткий лимит размера
(например, 50 МБ), путь вывода вида `_checked/<ГГГГ>/<студент>/`,
front-matter выключен или с дополнительными полями (`student`, `course`).

**Плюс:** не нужно держать в голове набор флагов на каждый прогон.

**Риск:** фактически — обёртка над тем, что уже есть. Может
размножить точки входа без реальной выгоды.

**Статус:** идея, не делать до явного запроса.

## 2. Убрать `.msg` (Outlook) из SUPPORTED_SUFFIXES

**Суть:** поддержка `.msg` через `markitdown[outlook]` тянет `olefile` и
`compressed-rtf`. Студенты редко шлют сами `.msg`-файлы — обычно
вложенные `.docx`/`.pdf`. Если формат не используется, его можно
выкинуть и облегчить дерево зависимостей.

**Плюс:** меньше зависимостей → меньше surface для SCA, проще
лицензионный gate, быстрее установка.

**Риск:** если сценарий «преподаватель проверяет почту» вдруг
появится — придётся возвращать.

**Статус:** идея, не делать до явного запроса. Если будешь
прорабатывать — сначала проверить, не используется ли `.msg` ни в
одном сценарии (LMS-выгрузки, ответы студентов по e-mail и т.п.).

## 3. Сохранение структуры папок при `-o` (флаг `--mirror`)

**Суть:** сейчас `tomd C:\students\*.docx -o C:\md` складывает все
`.md` плоско в `C:\md`. Если в `C:\students\ivanov\`,
`C:\students\petrov\`, `C:\students\sidorov\` лежат одноимённые
`курсовая.docx`, на выходе получается `курсовая.md`,
`курсовая (2).md`, `курсовая (3).md` — без следов, чьё.

Флаг `--mirror` (или `--preserve-tree`) при `-o` сохраняет
относительную структуру: `C:\md\ivanov\курсовая.md`,
`C:\md\petrov\курсовая.md` и т.д.

**Плюс:** для сценария «пачка студенческих работ → одна база знаний»
это решает реальную проблему идентификации.

**Решение по совместимости:** плоский вывод оставлен поведением по
умолчанию; структура включается только явным флагом `--mirror` /
`--preserve-tree`.

**Объём:** средняя фича — правки в `_plan_file_target()` (учёт
относительного пути), новая опция в argparse, тест на коллизию имён
из разных подпапок. Не косметика, но и не переписывание архитектуры.

**Статус:** реализовано в v1.2.0 как `--mirror` / `--preserve-tree`.
Плоский `-o` оставлен поведением по умолчанию для обратной совместимости.

## 4. PDF-OCR для сканов

**Суть:** студенты иногда присылают методички и старые учебные
материалы в виде «картинок внутри PDF» (сканы, факсы, сохранённые
страницы из старых книг). MarkItDown извлекает только текстовый слой —
если его нет, результат пустой или состоит из мусорных символов.

Подключить опциональный OCR-проход: если после извлечения текстового
слоя страница дала слишком мало символов на пиксель (или PDF помечен
как «image-only»), прогонять страницу через OCR и заменять
текстовый слой на распознанный.

**Кандидаты на реализацию:**

- `pytesseract` (обёртка над Tesseract). Самое простое, но Tesseract
  отдельно ставить (`choco install tesseract` / `apt install
  tesseract-ocr` / `brew install tesseract`). Для русского — языковой
  пакет `rus`. Качество на русском — среднее.
- `ocrmypdf` (обёртка, сам зовёт Tesseract + Ghostscript). Удобно:
  принимает PDF, возвращает PDF с текстовым слоем. Дальше тот же
  MarkItDown. Плюс — оптимизация, deskew, очистка. Минус — те же
  внешние зависимости.
- `paddleocr` (PaddlePaddle). Лучше русский, но тянет ~500 МБ
  зависимостей и тяжёлый рантайм. Для CLI-утилиты, которая гордится
  «поставил и работает» — избыточно.
- Облачные API (Google Vision, Azure Read, Yandex Vision). Лучше
  качество распознавания, но требует ключей, сетевых вызовов и сразу
  убивает сценарий «обработка недоверенных студенческих файлов без
  утечки в облако».

**Объём работ (для варианта с `ocrmypdf`):**

1. Опциональная зависимость в `pyproject.toml` (через `[pdf-ocr]`
   extra, чтобы не тянуть по умолчанию).
2. Проверка наличия `ocrmypdf` и `tesseract` в PATH; понятная
   диагностика, если их нет.
3. Новый флаг `--ocr {auto,off,always}` (по умолчанию `auto` —
   распознаём только image-only страницы).
4. Препроцесс: после `MarkItDown.convert()` померить объём
   извлечённого текста на страницу; если меньше порога — прогнать PDF
   через `ocrmypdf --language rus+eng` во временный файл, сконвертить
   его, удалить временный файл.
5. В sandbox-режиме subprocess с тем же timeout, что и для обычной
   конвертации. Для больших сканов OCR может занимать десятки секунд
   на страницу — `--conversion-timeout` придётся поднимать или
   вводить отдельный `--ocr-timeout`.
6. Тесты: пустой PDF (text-only) не должен трогать OCR; PDF-скан
   должен пройти через OCR; отсутствие `tesseract` в PATH должно давать
   внятную ошибку, а не падать в непонятном месте.
7. CI: добавить установку `tesseract-ocr` + `tesseract-ocr-rus` в job
   smoke, иначе smoke-тест на OCR не сможет прогоняться на ubuntu.
8. SBOM: `ocrmypdf` тянет за собой `pikepdf`, `reportlab`,
   `pdfminer.six`, `tesserocr` — все под MIT/Apache, но новые пакеты
   надо прогнать через license gate.

**Реалистичная оценка:** 1–2 дня работы (включая тесты и шлифовку
диагностики). Без учёта установки `tesseract` на машинах пользователя
(это отдельная инструкция в README).

**Риск:**

- `tesseract` — внешняя зависимость, не Python-пакет. Установщик
  `install.ps1` на Windows должен будет её подтянуть (`winget install
  tesseract` + языковой пакет) или хотя бы подсказать.
- На macOS/Linux пользователю нужно доставить `tesseract-ocr` самому.
  Это рушит принцип «поставил и работает».
- OCR-качество на рукописных/старых сканах — слабое, можно получить
  искажённый Markdown и не понять, что проблема в распознавании, а не
  в исходнике.
- На больших сканах (200+ страниц) время обработки вырастает
  значительно. Студенты редко шлют такие, но если пришлют — нужен
  понятный прогресс, а не «висит N минут».

**Статус:** идея, **самая проработанная из четырёх** — кандидат №1
на реализацию, если в сценарии преподавателя это реально мешает.
Перед началом — проверить, как часто приходят именно сканы (а не
нормальные PDF с текстовым слоем), и попробовать `ocrmypdf` руками на
паре реальных файлов: если качество распознавания устраивает — браться.
