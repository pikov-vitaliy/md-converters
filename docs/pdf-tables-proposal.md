# Предложение: корректный перенос PDF-таблиц в Markdown

Статус: **РЕАЛИЗОВАНО** (вариант A+B), 2026-06-27. Изначально — исследование
+ варианты; по выбору пользователя реализован вариант A+B в
`convert_to_md.py` (функции `_pdf_tables_result`, `_page_blocks`,
`_segment_table_rows`, `_join_continued_tables`, `_table_to_gfm`, флаг
`--pdf-tables`, поле `pdf_tables: N`). Дополнительно по обратной связи:
(1) исправлена регрессия — проза вокруг таблиц больше не схлопывается в
одну строку (`_row_text` сохраняет переносы); (2) добавлена конвертация
псевдографики таблиц (`│┌┬┐…`) в GFM (`_convert_box_tables` в `tidy`).
Тесты на эту работу — в `tests/test_pdf_tables.py` (всего по проекту 100+
pytest-тестов). Документ ниже сохранён как обоснование
выбора и обзор кандидатов. Тестовый кейс:
`Предзащита_ВКР_Липатова_итог_и_памятка.pdf` (21 стр., реальный документ,
7 геометрических таблиц + 1 псевдографика).

Известный edge (не блокер): строка таблицы, разорванная между страницами
ПОСРЕДИ ячейки (+ повтор заголовка на новой странице), восстанавливается
не полностью — часть строк выходит прозой. Сегментация по строкам-склейкам
нужна для РАЗДЕЛЕНИЯ слипшихся таблиц в пределах страницы и конфликтует с
СОХРАНЕНИЕМ разорванной строки; выбран безопасный дефолт (разделять).
Пример: хвост «Приложение B» (строки «Статья»/«Финал») в тестовом файле.

## 1. TL;DR

- **Корень проблемы найден.** Таблицы «едут» не потому, что «PDF плохой», а
  потому что PDF-конвертер MarkItDown 0.1.6 распознаёт таблицы слабой
  эвристикой — кластеризацией **слов по X-координатам** — и **игнорирует
  геометрию таблицы** (линии/рамки/заливки). Из-за этого соседние колонки
  слипаются, широкие рвутся, проза «вмешивается» в таблицу, а в ячейках с
  переносом строк ломается вся разметка.
- **Решение есть и оно дешёвое.** `pdfplumber` — который **уже стоит** в
  окружении (его тянет `markitdown[pdf]`) — умеет находить таблицы по их
  геометрии (`find_tables` / `extract_tables`). На твоём реальном документе
  это даёт **корректные таблицы там, где сейчас каша**, и при этом **не
  добавляет ни одной новой зависимости** и ничего в SBOM/лицензии.
- **Рекомендация:** свой PDF-путь на `pdfplumber.find_tables` + фильтр «это
  правда таблица» + обрезка прилипшей прозы, с интерливингом текста и
  таблиц по порядку чтения. Тяжёлые/облачные/AGPL/torch-варианты не нужны.

## 2. Доказательство (до/после на твоём файле)

### Было — §3.1 «Чек-лист… в pptx» через текущий `tomd` (MarkItDown)
Фрагмент из `Предзащита_ВКР_Липатова_итог_и_памятка.md`: корректный
заголовок таблицы, а дальше — **повторные строки-разделители посреди
таблицы**, текст ячеек выброшен наружу, перепутан порядок столбцов:

```
| №   | Что | Где | Статус |
| --- | --- | --- | ------ |
Заменить Notes в слайде 1 — текст про АРМ банка,
| 1   |     | Слайд 1 | 🔴   |
| --- | --- | ------- | --- |
Пиков В.А.
2 Заменить Notes в слайде 3 — текст про НКК, ПК-1, 851-П Слайд 3 🔴
| 3 Заменить Notes в слайде 4 — список 5 задач |     | Слайд 4 | 🔴   |
...
```

### Стало — `pdfplumber.find_tables()` + фильтр + обрезка прозы
```
| № | Что | Где | Статус |
| --- | --- | --- | --- |
| 1 | Заменить Notes в слайде 1 — текст про АРМ банка, Пиков В.А. | Слайд 1 | 🔴 |
| 2 | Заменить Notes в слайде 3 — текст про НКК, ПК-1, 851-П | Слайд 3 | 🔴 |
| 3 | Заменить Notes в слайде 4 — список 5 задач | Слайд 4 | 🔴 |
| 4 | Убрать «p < 0,001» со слайда 16 | Слайд 16 | 🔴 |
| 5 | Разбить слайд 4 (задачи) на отдельный слайд | Слайд 4 | 🟡 |
...
| 11 | Опционально: «продающий» слайд… | Новый слайд | 🟢 |
```

То же подтвердилось на других таблицах документа: «Аспект | Оценка»,
«Момент | Фраза-якорь», «Ситуация | Фраза» — все извлеклись корректно как
2 колонки с эмодзи-статусами. Технический факт: в этом PDF **нет линий**
(`lines=0` на всех страницах) — таблицы безбордюрные, нарисованы
заливками-прямоугольниками; pdfplumber находит их по краям заливок.

### Важная деталь: нужен фильтр «это таблица»
Если просто отдать всё, что нашёл pdfplumber, на чисто текстовых страницах
(Q&A-памятка) он вернёт ложные «таблицы» из одного столбца (проза,
обёрнутая в таблицу). Поэтому в решении обязателен фильтр:
- ≥ 2 колонок и ≥ 2 строк с 2+ заполненными ячейками;
- не «сплошная проза» (доля длинных ячеек < 50%);
- обрезка крайних строк-склеек (заголовок над таблицей / абзац под ней,
  прилипшие из-за заливки);
- отбрасывание полностью пустых колонок.

С фильтром на тестовом файле: из 21 страницы **9 распознаны как таблицы,
12 — как текст** — это совпадает с реальной структурой документа.

### End-to-end сборка страницы (валидация «низкого риска»)
Извлечь таблицу — половина дела; вторая половина — **собрать всю страницу**
(проза + таблица по порядку чтения, без дублирования текста таблицы прозой).
Проверено на целых страницах (`assemble.py`, см. Приложение A):
- **стр.1 и стр.7** собираются идеально: заголовок над таблицей и абзац под
  ней выводятся как проза (переэмитятся из «прихваченных» в таблицу строк),
  между ними — чистая таблица; ключевая строка таблицы встречается в выводе
  **ровно один раз** (дублирования нет);
- текст вне таблицы берётся вырезанием полосы страницы вне bbox таблицы —
  поэтому он не дублирует содержимое таблицы.

**Честные краевые случаи** (стр.16 «План по дням»):
1. **лишний пустой столбец** — лечится отбрасыванием полностью пустых колонок;
2. **две логические таблицы слиплись в одну** — фоновые заливки идут
   непрерывно через границу разделов, и `find_tables` берёт их одним блоком;
   нужна разбивка по «строкам-заголовкам» (одна заполненная ячейка между
   блоками) или по разрывам высоты строк;
3. **проза между строками таблицы** (не сверху/снизу, а в середине) текущей
   обрезкой не убирается — редкий случай, лечится той же разбивкой.

Вывод: для «чистых» таблиц (большинство) решение даёт готовый результат и
низкий риск; для «слипшихся» страниц нужна доп. шлифовка (это уже объём
Варианта B). Поэтому честная оценка Варианта A — **S→M**, и основная работа
— именно сборка/разводка, а не извлечение (извлечение уже решено).

POC-скрипты (вне репозитория, в scratchpad сессии): `gen_pdf.py`, `poc.py`,
`diag_real.py`, `render_real.py`, `assemble.py`. Ключевой код — в Приложении A.

## 4. Почему именно так (root cause в коде MarkItDown)

`markitdown/converters/_pdf_converter.py` (0.1.6):
- `convert()` для каждой страницы зовёт `_extract_form_content_from_words()`,
  который работает на `page.extract_words()` — то есть **по координатам
  слов** с адаптивным допуском кластеризации. Реальная геометрия таблицы
  (линии/края прямоугольников/заливки) **не используется**.
- `page.extract_tables()` / `find_tables()` (которые учитывают геометрию)
  **не вызываются** вообще.
- Если на странице «форм-контент» не найден — откат на
  `pdfminer.high_level.extract_text()` (плоский текст).

Отсюда и слипание/разрыв колонок: допуск кластеризации слов либо склеивает
две узкие колонки, либо рвёт одну широкую; многострочные ячейки рушат
выравнивание; повторное «обнаружение таблицы» по кускам печатает новый
заголовок-разделитель в середине.

## 5. Варианты интеграции (предлагается выбрать)

Точка входа уже есть: проект и так особо обрабатывает `.pdf` в
`_convert_file_to_target` (подсчёт страниц `pypdfium2`, диагностика
текстового слоя). Туда же ложится новый PDF-путь.

### Вариант A — «Минимальный, целевой» (рекомендую) · трудозатраты S→M
Свой `_pdf_to_markdown(path)`:
1. по страницам: `page.find_tables(strategy)` → для каждой найденной
   таблицы фильтр «это таблица» + обрезка прозы + GFM-рендер
   (свой форматтер: экранировать `|`, схлопывать `\n` в ячейке,
   дополнять рваные строки, синтезировать строку-разделитель);
2. текст вне таблиц — как сейчас (`extract_text` / MarkItDown);
3. собрать текст и таблицы **по вертикали страницы** (порядок чтения);
4. если таблиц не найдено — поведение ровно как сейчас (никаких регрессий).
- Зависимости: **0 новых установленных** пакетов — pdfplumber уже стоит.
  НО: его нужно объявить **прямой** зависимостью в `pyproject.toml`
  (`pdfplumber>=0.11.9`), а не полагаться на транзитивную поставку через
  `markitdown[pdf]`: под нашим потолком `markitdown<1.0.0` состав extra
  может измениться и молча сломать импорт. То есть «0 новых установленных,
  +1 явная строка в pyproject».
- Чинит ровно твой кейс (стр.1/7 — идеально end-to-end). Риск низкий:
  есть фолбэк на текущий путь; «слипшиеся» страницы (стр.16) выходят не
  хуже текущего, а доводятся уже в Варианте B.

### Вариант B — «Надёжный» · трудозатраты M
A + : двойная стратегия (`lines`/edges И `text`) с выбором лучшей по
эвристике; склейка таблиц, разорванных между страницами (одинаковая
ширина/заголовок); флаг CLI `--pdf-tables auto|off` и поле
`pdf_tables: N` во front-matter. Чуть больше кода и тестов, заметно
устойчивее на разной вёрстке.

### Вариант C — «Максимум качества», опционально · трудозатраты L
Дополнительный extra (`pip install md-converters[tables-ml]`) на **Docling**
(MIT, модель TableFormer) для безбордюрных/сканированных таблиц, где
геометрии нет. Тянет torch — тяжело и рискованно на Python 3.14; включать
только при реальной потребности и только как явный opt-in пользователя.
Альтернативный opt-in — `[pymupdf]` (AGPL, не по умолчанию) ради
`pymupdf4llm` в одну строку.

## 6. Риски и подводные камни (предупреждён = вооружён)

1. **Безбордюрные/сканированные таблицы — предел геометрии.** pdfplumber
   силён на разлинованных и на заливках; на полностью «голой» сетке без
   линий и фона качество падает (это общий предел rule-based — здесь
   выигрывают только DL-модели, см. Вариант C). Для сканов (image-only
   PDF) таблиц нет в принципе — нужен OCR (проект это уже детектирует).
2. **Ложные таблицы из прозы.** Без фильтра §2.x проза обернётся в
   1-колоночную «таблицу». Фильтр обязателен (проверено).
3. **GFM не любит переносы в ячейках.** `\n` внутри ячейки нужно
   схлопывать в пробел или `<br>`; `|` — экранировать. Готовые
   `pandas.to_markdown`/`tabulate` **не экранируют `|`** (известный баг) —
   поэтому свой маленький форматтер, без новых тяжёлых зависимостей.
4. **Лимит строк ≤ 79 и стиль проекта** (E501) — соблюдать при правках.

## Приложение A. Заимствуемый код (проверен на твоём PDF)

Самодостаточно, без новых тяжёлых зависимостей (только уже стоящий
`pdfplumber`). Это рабочий каркас Варианта A: извлечение → фильтр → обрезка
с переэмитом прозы → GFM → сборка страницы по порядку чтения без дублей.

```python
import pdfplumber

STRAT = {"vertical_strategy": "lines", "horizontal_strategy": "lines"}


def _populated(row):
    return sum(1 for c in row if c and str(c).strip())


def _is_real_table(rows):
    """Отсекает прозу, обёрнутую в «таблицу»."""
    if not rows or max(len(r) for r in rows) < 2:
        return False
    if sum(1 for r in rows if _populated(r) >= 2) < 2:
        return False
    cells = [str(c) for r in rows for c in r if c and str(c).strip()]
    longish = sum(1 for c in cells if len(c.strip()) > 60)
    return not (cells and longish / len(cells) > 0.5)


def _trim_prose(rows):
    """-> (lead_text, clean_rows, trail_text): крайние строки-склейки
    (заполнена < 2 ячеек) возвращаются как проза, чтобы не потерять их."""
    rows = [list(r) for r in rows]

    def row_text(r):
        return " ".join(str(c).replace("\n", " ").strip()
                        for c in r if c and str(c).strip())
    lead, trail = [], []
    while rows and _populated(rows[0]) < 2:
        lead.append(row_text(rows.pop(0)))
    while rows and _populated(rows[-1]) < 2:
        trail.insert(0, row_text(rows.pop()))
    return ("\n".join(t for t in lead if t), rows,
            "\n".join(t for t in trail if t))


def _gfm(rows):
    """Список строк -> GitHub-flavored Markdown (экранирует |, схлопывает \\n,
    дополняет рваные строки). pandas/tabulate НЕ используем: они не
    экранируют | (issue tabulate#241) и тянут лишние зависимости."""
    width = max(len(r) for r in rows)
    norm = []
    for r in rows:
        cells = [("" if c is None else str(c)).replace("\n", " ")
                 .replace("|", "\\|").strip() for c in r]
        cells += [""] * (width - len(cells))
        norm.append(cells)
    out = ["| " + " | ".join(norm[0]) + " |",
           "| " + " | ".join("---" for _ in norm[0]) + " |"]
    out += ["| " + " | ".join(r) + " |" for r in norm[1:]]
    return "\n".join(out)


def _band_text(page, top0, top1):
    if top1 - top0 < 2:
        return ""
    crop = page.crop((0, max(0, top0), page.width,
                      min(page.height, top1)))
    return (crop.extract_text() or "").strip()


def page_to_md(page):
    """Целая страница -> Markdown: проза и таблицы по порядку чтения,
    без дублирования (текст таблицы не повторяется прозой)."""
    tables = [t for t in page.find_tables(STRAT)
              if _is_real_table(t.extract())]
    if not tables:
        return (page.extract_text() or "").strip()
    tables.sort(key=lambda t: t.bbox[1])
    parts, cursor = [], 0.0
    for t in tables:
        _, top, _, bottom = t.bbox
        above = _band_text(page, cursor, top)
        if above:
            parts.append(above)
        lead, rows, trail = _trim_prose(t.extract())
        if lead:
            parts.append(lead)
        if rows:
            parts.append(_gfm(rows))
        if trail:
            parts.append(trail)
        cursor = bottom
    below = _band_text(page, cursor, page.height)
    if below:
        parts.append(below)
    return "\n\n".join(parts)
```
