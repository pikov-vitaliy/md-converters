# CLAUDE.md — контекст проекта для AI-агентов

Этот файл — онбординг для Claude Code и любых AI-агентов. Прочитай его
первым, чтобы сразу быть в контексте проекта. Язык проекта — русский
(интерфейс утилит, README, комментарии). Пользователь: Виталий Пиков
(vitaliy@pikov.com), сфера — DevSecOps / безопасность; конвертирует
отчёты инструментов (SBOM, уязвимости, ГОСТ/РБПО-аудит) в Markdown для
базы знаний.

## Что это

**md-converters** — утилита перевода документов в Markdown на базе
[MarkItDown](https://github.com/microsoft/markitdown) (Microsoft). Одна
команда `tomd` определяет формат по расширению и конвертирует: PDF, HTML,
Word (`.docx`), Excel (`.xlsx`), PowerPoint (`.pptx`), CSV, JSON, XML,
EPUB, Outlook `.msg`, Jupyter, RSS и веб-страницы по URL.

## Архитектура

- **`convert_to_md.py`** — единственный скрипт, вся логика. Раньше было два
  отдельных (`convert_pdf_to_md.py`, `convert_html_to_md.py`) — они УДАЛЕНЫ
  и слиты сюда.
- Команды-обёртки: `tomd` = весь функционал; `pdf2md` = `tomd --only pdf`;
  `html2md` = `tomd --only html,htm`. Различие только в фильтре формата при
  обходе папки/маски (одиночный явный файл конвертируется всегда).
- Точки входа для pip (console_scripts в `pyproject.toml`):
  `cli_tomd` / `cli_pdf` / `cli_html` в `convert_to_md.py`.
- Поле `generator` во front-matter выводится из набора расширений
  (`_tool_name`): `{.pdf}`→`pdf2md`, `{.html,.htm}`→`html2md`, иначе `tomd`.

## Возможности / флаги

`-r` рекурсия (пропускает `node_modules/.git/.next/dist/build/venv/...`);
`-f`/`--force` перезапись (иначе существующие `.md` пропускаются);
`-o DIR` вывод в одну папку; совпадения имён `.md` всегда (и без `-o`)
разводятся суффиксами `(2), (3)...` со сверкой `source:` во front-matter
уже лежащих на диске файлов;
`--only EXT[,EXT]` фильтр форматов при маске/папке;
`--pdf-tables auto|off` (по умолчанию `auto`) — извлечение таблиц из PDF
по геометрии через pdfplumber; `off` — штатный текстовый путь MarkItDown;
`--keep-images` оставить base64-картинки (по умолчанию сворачиваются в
плейсхолдер `![...]()`); `--no-frontmatter` без YAML-шапки;
URL (`http(s)://`) — тянет страницу через `convert_url`.
Прочее: маски, папки, front-matter (title/source/converted/generator),
чистка вывода (`tidy`), сводка с перечнем ошибок, автоопределение кодировки
HTML (cp1251 и др.) через перекодировку во временный UTF-8.

## Известные ограничения (не баги)

- **PDF-таблицы — извлекаются через pdfplumber по геометрии** (линии/края
  заливок), а НЕ штатным путём MarkItDown (тот ищет таблицы по координатам
  слов и слипает/рвёт колонки). См. функции `_pdf_tables_result`,
  `_page_blocks`, `_segment_table_rows`, `_join_continued_tables`,
  `_table_to_gfm` в `convert_to_md.py`. Есть фильтр «это таблица» (отсекает
  прозу), обрезка/переэмит прилипшей прозы, склейка переносных таблиц по
  повтору заголовка, отброс пустых колонок, сборка текста и таблиц по
  порядку чтения без дублирования. Поле `pdf_tables: N` во front-matter.
  Отключение — `--pdf-tables off`. Предел метода: полностью «голые»
  таблицы без линий/фона и сканы (нет геометрии) — там best-effort, нужен
  ML-извлекатель (Docling и т.п. — отвергнут как обязательная зависимость:
  torch/вес/риск на Python 3.14). PyMuPDF/pymupdf4llm не берём — AGPL.
  Обоснование выбора и обзор кандидатов — в `docs/pdf-tables-proposal.md`.
- **Псевдографика таблиц (box-drawing `│┌┬┐├┼┤└┴┘─`) -> GFM.** Такие
  «таблицы» нарисованы символами рамок прямо в тексте (часто — генераторы/
  ИИ), геометрии у них нет, и в Markdown-вьюере они разъезжаются. `tidy()`
  (`_convert_box_tables`/`_box_block_to_gfm`) конвертирует их в GFM:
  двухстрочные шапки сливаются, внешние рамки срезаются. Работает для
  любого формата; триггерится только при наличии вертикалей `│`/`┃` (на
  прозу и на готовый GFM с ASCII `|` не влияет).
- **Чистка PDF-текста (`_clean_pdf_text` в `_pdf_tables_result`).** На
  многостраничных PDF (≥3 стр.) убираются сквозные колонтитулы
  (повторяющиеся ≥ `page_count//2` короткие строки — номер документа,
  версия) и страничные номера (только если колонтитул найден). Плюс
  экранируются ложные ATX-заголовки: в PDF нет настоящих `#`-заголовков,
  поэтому `# ...` (комментарий конфига и т.п.) рендерился бы как H1 —
  экранируем в `\#`. Строки таблиц (`|`) не трогаются. Уникальные
  документы без повторов не страдают (одиночное число не режется, если
  колонтитул не обнаружен).
- **Фенсинг кода/команд/конфигов (`_fence_code_blocks`/`_classify_code_line`
  в `_clean_pdf_text`).** Прогоны строк-кода (команды, SQL, конфиги, вывод)
  оборачиваются в ``` — иначе `<плейсхолдеры>` съедаются рендером как HTML,
  `#`-комменты становятся заголовками, `|-vda` ломается как таблица.
  Классификатор построчный: русская проза кириллице-доминантна → проза;
  код — латиница-доминантная строка / команда (sudo, CLI `cmd -flag`) / SQL
  (CREATE/SELECT/MAC…) / `$VAR` / `{<...>}` / `UUID=`/`deb ` / продолжение
  `\`. Настоящие таблицы (`| … |`) и псевдографику (`│`) не трогаем (они —
  границы блока). `#`-экранирование делается только ВНЕ фенсов. Порог
  настроен на 0 ложных срабатываний на корпусе из 10 реальных документов
  Astra (3371 код-фенс, прозы во фенсах нет). Код построчно перемешан с
  прозой, поэтому фенсы часто короткие — это норма.
- **Только стратегия `lines` для таблиц** (`_PDF_TABLE_STRATEGIES`).
  Стратегия `text` (кластеризация слов) убрана: на плотном коде/SQL и
  многоколоночной прозе она порождала фейковые мега-«таблицы» (76x7) и
  крошила текст в огрызки. Настоящие таблицы разлинованы/с заливкой → их
  берёт `lines` (через края прямоугольников).
- **Известный edge:** строка таблицы, разорванная между страницами ПОСРЕДИ
  ячейки (метка на одной стр., значение — на следующей, плюс повтор
  заголовка), реконструируется не полностью (часть строк выходит прозой).
  Сегментация по строкам-склейкам нужна, чтобы РАЗДЕЛЯТЬ слипшиеся таблицы
  (одна страница), и это конфликтует с СОХРАНЕНИЕМ разорванной строки —
  выбран безопасный дефолт (разделять). Пример: хвост «Приложение B» в
  тестовом документе (строки «Статья»/«Финал»). Всё равно сильно лучше
  штатного MarkItDown.
- **Word-таблицы:** MarkItDown даёт пустую верхнюю строку-заголовок, реальные
  заголовки уезжают в первую строку тела (данные не теряются).
- **Картинки:** base64 по умолчанию сворачиваются в плейсхолдер (решение
  пользователя — так и оставить).
- **PPTX-картинки:** MarkItDown не извлекает файлы картинок из `.pptx` —
  пишет битую ссылку `![alt](ИмяФигуры.jpg)` (а при пустых именах фигур,
  напр. из R7-Office, вырожденную `![](.jpg)`); просмотрщики Markdown
  рисуют вместо неё ошибку `EntryNotFound`/`ENOENT`. `tidy()` сворачивает
  такие ссылки в плейсхолдер (`_PHANTOM_IMG`, только для `.pptx`).
- **Управляющие символы:** перенос строки внутри абзаца PowerPoint
  (`<a:br/>`) python-pptx отдаёт как vertical tab `\x0b` — в `.md` это
  «квадратик». `tidy()` чистит: `_CTRL_TO_SPACE` (разделители строк →
  пробел) и `_CTRL_DROP` (C0/DEL/soft hyphen/ZWSP/BOM — удалить).

## Подводные камни для агентов

- **Кириллица в имени `.lnk`:** `WScript.Shell.CreateShortcut` теряет
  кириллицу в ИМЕНИ файла (downconvert в ANSI → `?` → невалидное имя).
  Обход в install.ps1: создать ярлык с латинским именем, затем
  `Move-Item`/`Rename-Item` в кириллицу (это Unicode). Не «чини» обратно.
- **Длина строк в `convert_to_md.py`:** держи ≤ 79 символов — IDE-линтер
  ругается E501. Проверяй после правок.
- **`.ps1` без BOM:** дочерний `pwsh -File` читает как UTF-8 — кириллица ок;
  но профиль пишем с `utf8BOM`, чтобы читали и PowerShell 5.1, и 7.
- **Песочница** периодически блокирует `Remove-Item` с маской `\*` или
  подозрительными путями — используй `-LiteralPath` на конкретную папку.

## Как разрабатывать и проверять

1. Правки — в `V:\md-converters\convert_to_md.py` (и обёртки/README).
2. Проверка: компиляция `python -m py_compile convert_to_md.py`; длина строк
   ≤ 79; функциональный тест на реальных файлах во временной папке
   (`$env:TEMP`), результат смотреть, temp убирать.
3. Установщик идемпотентен — `install.ps1` можно гонять повторно (сам
   убирает старый блок профиля и пункт Send To, прописывает заново).
4. CI (GitHub Actions) на каждый push в `main` гоняет ruff-линт и
   смоук-тесты (HTML, CSV, пропуск без `--force`, ошибки флагов) на
   ubuntu (Python 3.10/3.11/3.12/3.13/3.14) и windows (3.12/3.14) —
   должен быть зелёным.

   **CI активен; репозиторий публичный.** 2026-06-16 репозиторий
   сделан публичным (`gh repo edit --visibility public`), потому что
   на приватном репо Actions тратят платные минуты, а платёж/лимит
   тогда исчерпался — CI падал мгновенно с «recent account payments
   have failed» и спамил «CI: All jobs have failed» на каждый push.
   На публичном репо стандартные раннеры бесплатны без лимита, биллинг
   больше не помеха. Если когда-нибудь понадобится пауза без оплаты —
   `gh workflow disable CI` (→ `disabled_manually`), вернуть —
   `gh workflow enable CI`.
   Локально тот же набор, что и в CI (голый `python` на PATH —
   заглушка Microsoft Store, поэтому через `uv`):
   `uv lock --check && uv sync --frozen && uv run --frozen ruff check
   convert_to_md.py tests tools && uv run --frozen pytest -q`.
