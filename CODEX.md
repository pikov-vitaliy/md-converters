# CODEX.md - план доведения md-converters до качественной утилиты

Дата фиксации плана: 2026-06-13.

## Роль Codex в этом проекте

Я работаю как инженер по безопасной разработке, ревьюер и исполнитель
доработок для `md-converters`.

Моя задача - не угадывать и не "косметически улучшать" проект, а довести его
до состояния надежной локальной утилиты для быстрого перевода документов,
отчетов, учебных материалов и материалов по ИБ в Markdown.

Приоритеты:

1. Безопасность обработки недоверенных файлов и URL.
2. Корректность результата и отсутствие потери пользовательских данных.
3. Воспроизводимость установки и зависимостей.
4. Понятное поведение CLI без скрытых сюрпризов.
5. Проверяемое качество через тесты, CI, SCA и контрольные сценарии.

Ориентиры: GOST R 56939-2024, требования ФСТЭК к безопасной разработке,
NIST SSDF, ISO/IEC 27001, ISO/IEC 27034, OWASP, CWE, SLSA, CycloneDX.

## Цель продукта

`md-converters` должен быть удобной и безопасной персональной утилитой для
конвертации PDF, HTML, DOCX, XLSX, PPTX, CSV, JSON, XML, EPUB, MSG, IPYNB,
RSS и веб-страниц в Markdown.

Целевое качество:

- пользовательские `.md` не перезаписываются неожиданно;
- одноименные источники из разных папок корректно сопоставляются с
  результатами;
- русскоязычные HTML в UTF-8, Windows-1251 и близких кодировках
  обрабатываются предсказуемо;
- выходной Markdown безопасен для просмотра в Obsidian, VS Code preview,
  статических сайтах и LMS;
- URL-режим не превращается в SSRF/DoS-инструмент;
- зависимости фиксируются, проверяются на уязвимости и лицензии;
- регрессы ловятся тестами до ручного использования.

## Проверенное состояние на момент плана

Что уже хорошо:

- основной код сосредоточен в `convert_to_md.py`;
- прямых `eval`, `exec`, `pickle`, shell-вызовов в Python-части не найдено;
- защита от перезаписи по умолчанию есть;
- `markitdown` ограничен диапазоном `<1.0.0`;
- CI уже проверяет Linux Python 3.10/3.11/3.12/3.13 и Windows
  Python 3.12/3.14;
- `ruff check convert_to_md.py` проходил без ошибок;
- smoke HTML -> Markdown проходил во временную папку;
- быстрый `pip-audit` по текущей установленной цепочке MarkItDown на
  2026-06-12 не нашел известных уязвимостей.

Ограничение этой оценки:

- без lockfile/SBOM результат SCA относится к текущей среде, а не ко всем
  будущим установкам.

## Подтвержденные недостатки

### 1. URL-режим без собственной сетевой политики

`convert_url()` передает URL в `MarkItDown.convert_url()`. В установленной
версии MarkItDown HTTP-ответ загружается через `requests` без явного timeout
и затем читается в память целиком.

Риски:

- SSRF к `localhost`, private/link-local адресам и внутренним сервисам;
- зависание на медленных ответах;
- исчерпание памяти большим ответом;
- непредсказуемое поведение редиректов.

Стандарты и практики: CWE-918, CWE-400, OWASP SSRF Prevention Cheat Sheet,
NIST SSDF PW.8/RV.1.

### 2. Stored content injection в выходной Markdown

Вредоносный HTML может протащить в Markdown активные ссылки или опасную
разметку, например через `alt` у изображения.

Риск не в выполнении кода самим конвертером, а в следующем этапе:
просмотр Markdown в Obsidian, VS Code preview, статическом сайте, LMS или
другой системе может активировать небезопасный контент.

Стандарты и практики: CWE-79, CWE-116, OWASP XSS Prevention Cheat Sheet.

### 3. Нет ресурсных лимитов и изоляции парсеров документов

PDF, DOCX, XLSX, PPTX, MSG и EPUB обрабатываются тяжелыми сторонними
парсерами в основном пользовательском процессе.

Риски:

- зависание на сложном или поврежденном документе;
- большое потребление памяти;
- эксплуатация уязвимости в стороннем парсере с правами пользователя;
- обработка файлов от студентов, подрядчиков или внешних систем без
  defense-in-depth.

Стандарты и практики: CWE-400, NIST SSDF PW.4/PW.8, ISO/IEC 27034.

### 4. Одноименные файлы из разных папок плохо сопоставляются при `-o`

При выводе в одну папку `source:` сейчас хранит только имя файла. Если есть
`a/report.html` и `b/report.html`, повторный запуск только одного файла может
попасть не в тот `.md`.

Риск:

- перезапись или обновление не того результата;
- потеря доверия к пакетной конвертации больших деревьев документов.

### 5. Автоопределение Windows-1251 ненадежно

Короткий HTML в Windows-1251 может быть ошибочно распознан как cp1250.

Риск:

- русскоязычный текст превращается в нечитаемую строку;
- README обещает более надежное поведение, чем фактически гарантирует код.

### 6. Ручной парсер аргументов принимает флаги как значения

Примеры проблемного поведения:

- `tomd file.html -o --force` может считать `--force` папкой вывода;
- `tomd file.html --only --force` может считать `--force` расширением.

Риск:

- неожиданная структура вывода;
- тихие ошибки в пакетных сценариях;
- сложность дальнейшего развития CLI.

### 7. Supply chain не воспроизводим

Есть диапазон зависимости, но нет lockfile, SBOM, hash-pinning и регулярной
проверки advisory/license в CI.

Риск:

- разные машины получают разные версии транзитивных зависимостей;
- сложно доказать состав поставки;
- сложно вести SCA/лицензионный контроль.

Стандарты и практики: NIST SSDF PS.3/RV.1, SLSA, CycloneDX,
ISO/IEC 27001 A.8.8.

## План устранения

### Этап 1. CLI и предотвращение потери данных

Цель: убрать тихие ошибки и сделать поведение пакетной конвертации
предсказуемым.

Работы:

1. Заменить ручной `_parse()` на `argparse`.
2. Явно запретить использование флагов как значений для `-o` и `--only`.
3. Валидировать `--only` строгим правилом: только расширения формата
   `pdf`, `.pdf`, `docx,xlsx`; запретить `/`, `\`, `:`, пробелы,
   управляющие символы и значения, начинающиеся с `-`.
4. Добавить в front matter стабильные поля:
   - `source_name`;
   - `source_path`;
   - `source_id`.
5. Для `source_id` использовать нормализованный абсолютный путь или
   относительный путь от корня сканирования плюс короткий хэш.
6. Исправить `_plan_target()`, чтобы при повторном запуске с `-o` файл
   попадал в свой прежний `.md`, а не в первый одноименный результат.

Критерии приемки:

- `tomd file.html -o --force` завершается ошибкой разбора аргументов;
- `tomd file.html --only --force` завершается ошибкой разбора аргументов;
- `a/report.html` и `b/report.html` при `-o out` получают разные `.md`;
- повторный запуск только `b/report.html -o out --force` обновляет именно
  результат `b/report.html`;
- старые `.md` без новых полей не ломают работу.

### Этап 2. Безопасный Markdown по умолчанию

Цель: результат можно безопаснее открывать в Markdown-просмотрщиках и
публиковать после дополнительной редакторской проверки.

Работы:

1. Добавить функцию sanitization для выходного Markdown.
2. Экранировать `alt` и другие вставляемые фрагменты Markdown.
3. Нейтрализовать ссылки со схемами:
   - `javascript:`;
   - `vbscript:`;
   - `file:`;
   - опасные `data:` кроме явно разрешенных изображений при `--keep-images`.
4. Удалять или экранировать raw HTML с обработчиками событий (`onerror`,
   `onclick` и т.п.).
5. Ввести флаг `--unsafe-raw-markdown` только для случаев, когда нужно
   сохранить исходную разметку без очистки.
6. Описать в README, что безопасная очистка снижает риск, но не заменяет
   проверку при публикации наружу.

Критерии приемки:

- тест с `javascript:` в HTML не оставляет активную ссылку в `.md`;
- тест с вредным `alt` не ломает Markdown-разметку;
- обычные таблицы, списки и заголовки не деградируют;
- `--keep-images` сохраняет ожидаемые изображения, но не включает опасные
  схемы без явного unsafe-режима.

### Этап 3. Безопасный URL-режим

Цель: URL-конвертация должна быть удобной, но не должна ходить во внутренние
ресурсы или бесконечно читать поток.

Работы:

1. Реализовать собственную загрузку URL до передачи данных в MarkItDown.
2. Разрешить только `http` и `https`.
3. Добавить timeout соединения и чтения.
4. Добавить лимит размера ответа, например 25-50 МБ по умолчанию.
5. Проверять DNS-результат и финальный адрес после редиректов.
6. По умолчанию запрещать:
   - `localhost`;
   - `127.0.0.0/8`;
   - `::1`;
   - RFC1918 private ranges;
   - link-local `169.254.0.0/16`;
   - multicast и unspecified адреса.
7. Добавить флаг `--allow-private-url` для осознанного локального сценария.
8. Добавить понятные ошибки: timeout, private address denied, response too
   large, unsupported scheme.

Критерии приемки:

- `tomd http://127.0.0.1:...` запрещен по умолчанию;
- `--allow-private-url` явно разрешает локальный адрес;
- большой ответ прерывается до исчерпания памяти;
- медленный ответ завершается timeout;
- редирект на private IP блокируется.

### Этап 4. Кодировки HTML

Цель: русскоязычные HTML из старых Windows-инструментов конвертируются
предсказуемо.

Работы:

1. Сначала определять BOM.
2. Затем читать `<meta charset=...>` и HTTP-equivalent meta.
3. Затем применять `charset_normalizer`.
4. Для коротких текстов с кириллицей добавить приоритетную проверку cp1251.
5. В диагностике писать фактически выбранную кодировку.
6. Добавить тесты для UTF-8, UTF-8 BOM, cp1251, koi8-r и некорректной
   кодировки.

Критерии приемки:

- короткий cp1251 HTML с `Привет` дает читаемый Markdown;
- UTF-8 не регрессирует;
- при неопределенной кодировке поведение явно диагностируется.

### Этап 5. Ресурсные лимиты и изоляция парсеров

Цель: недоверенный документ не должен бесконечно держать процесс или
потреблять память без контроля.

Работы:

1. Добавить параметры:
   - `--max-input-mb`;
   - `--timeout`;
   - `--no-sandbox` или обратный `--sandbox`, в зависимости от выбранной
     реализации.
2. Перед конвертацией проверять размер локального файла.
3. Запускать конвертацию каждого файла в отдельном subprocess с timeout.
4. Возвращать структурированный статус: ok, skip, fail, timeout, too_large.
5. На Windows по возможности использовать отдельную временную папку для
   каждого задания.
6. Не включать сетевой доступ в обработку локальных файлов.

Критерии приемки:

- большой файл отклоняется до передачи в парсер;
- зависший парсер убивается по timeout;
- пакетная обработка продолжает следующие файлы после timeout/fail;
- временные файлы очищаются.

### Этап 6. Supply chain, SBOM и лицензии

Цель: установка и состав зависимостей воспроизводимы и проверяемы.

Работы:

1. Добавить `uv.lock` или другой lockfile для разработки и CI.
2. Сформировать CycloneDX SBOM для Python-зависимостей.
3. Добавить `pip-audit` в CI.
4. Добавить проверку лицензий зависимостей.
5. Настроить Dependabot или Renovate для контролируемых обновлений.
6. Зафиксировать в README команду обновления lockfile/SBOM.
7. Хранить в репозитории воспроизводимые исходники контроля поставки:
   `uv.lock`, `tools/`, CI workflow и Dependabot config. Одноразовые SBOM и
   license reports формировать командой или CI artifact, чтобы не плодить
   timestamp/UUID-шум в истории.

Критерии приемки:

- CI падает при известной уязвимости в dependency graph;
- SBOM генерируется воспроизводимо;
- лицензии транзитивных зависимостей видны и проверяемы;
- установка на чистой машине повторяет ожидаемые версии.

### Этап 7. Тесты и документация

Цель: каждый исправленный риск получает тест и пользовательское описание.

Работы:

1. Ввести `pytest`.
2. Разнести тесты по группам:
   - CLI parsing;
   - output planning;
   - HTML encoding;
   - Markdown sanitization;
   - URL policy;
   - overwrite/skip behavior.
3. Оставить smoke-тесты CI, но дополнить unit-тестами.
4. Обновить README:
   - безопасный режим Markdown;
   - ограничения URL;
   - лимиты файлов;
   - работа с недоверенными документами;
   - SCA/SBOM.
5. Добавить короткий раздел "Security model".

Критерии приемки:

- `pytest` проходит локально;
- `ruff check` проходит;
- `py_compile` проходит;
- CI покрывает новые сценарии;
- README не обещает того, что не проверено тестами.

## Порядок реализации

Работать маленькими проверяемыми итерациями:

1. CLI parser + tests.
2. `source_path/source_id` + collision tests.
3. Markdown sanitization + adversarial HTML tests.
4. HTML encoding fix + cp1251 tests.
5. URL policy + SSRF/timeout/size tests.
6. subprocess timeout/size limits for local files.
7. lockfile, SBOM, pip-audit, license check.
8. README/security model update.
9. Финальный полный прогон.

После каждого этапа:

1. `ruff check convert_to_md.py`
2. `python -m py_compile convert_to_md.py` или `py -3.14 -m py_compile`
   на Windows, если `python` указывает на Microsoft Store stub.
3. `pytest`
4. Smoke-конвертация примеров во временную папку.
5. `git status --short`

## Definition of Done

Проект можно считать доведенным до целевого качества, когда:

- все подтвержденные недостатки из этого файла закрыты кодом и тестами;
- опасные URL блокируются по умолчанию;
- опасная Markdown-разметка нейтрализуется по умолчанию;
- одноименные файлы из разных папок не перетирают друг друга;
- русскоязычные HTML в cp1251 проходят тест;
- большие/зависшие документы не ломают пакетную обработку;
- CI включает lint, unit tests, smoke tests, SCA;
- SBOM и лицензии зависимостей можно воспроизвести;
- README точно описывает фактическое поведение.

## Принцип работы

Не принимать изменения "на глаз".

Каждое исправление должно иметь:

- конкретный риск или дефект;
- минимальный scoped patch;
- тест на старое плохое поведение;
- тест или smoke на нормальный пользовательский сценарий;
- понятное сообщение об ошибке для пользователя;
- отсутствие регрессий по существующим примерам.
